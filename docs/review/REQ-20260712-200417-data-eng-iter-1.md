# Core foundation review — iteration 1

Verdict: **FAIL — blocker fix required**.

## Finding

- **Blocker: an undone run can regain an included occurrence.**
  `CoreStore.add_source_record` and `CoreStore.add_occurrence` accept an already
  undone `run_id` (`backend/persistence.py:209` and
  `backend/persistence.py:281`). A later `undo_import_run` returns immediately
  when the run state is `undone` (`backend/persistence.py:385`), so the newly
  added occurrence remains `included`. This violates the acceptance criterion
  that every occurrence from the selected run becomes excluded, and violates
  INV-4's independently recoverable run-unit guarantee.

## Three-category review

- Unmet/partial criteria: atomic state-based undo is partial under the misuse
  sequence above.
- Scope creep: none; all changed files are inside the declared scope or the
  protocol-mandated request artifacts.
- Looks-done-but-wrong: the declared suite and completion gate are green, but
  they do not exercise writes after a run becomes undone.

Ease of misuse: a caller can invoke `undo_import_run(run_id)`, then
`add_source_record(run_id, ...)` and `add_occurrence(run_id, ...)`, and receive
successful IDs while creating an `included` occurrence under an `undone` run.
