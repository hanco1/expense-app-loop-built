## Blocker finding

### B1 - public persistence API bypasses component validation

`CoreStore` is publicly exported at `backend/__init__.py:4` and
`backend/__init__.py:7`. Its public `add_duplicate_decision()` method at
`backend/persistence.py:960` validates only the selected pair, then appends the
decision directly. The centralized projection is called only by the service
path before the write at `backend/analysis.py:215`; the projection itself is at
`backend/analysis.py:429`.

The review-owned probe creates valid `AB:A` and `BC:B` decisions and then calls
the exported store method for cycle-closing `AC:C`. Expected behavior is
`ValueError` before persistence and no history delta. Actual behavior appends
the third decision (`duplicate_decisions` 2 -> 3); the invalid committed state
is detected only later when an analysis read raises `RuntimeError: same_transaction
component must have exactly one structural keeper`.

This violates the criteria that every committed same component has exactly one
structural keeper and that rejected zero/multi-keeper proposals append no
history. It is also the standing ease-of-misuse path: a normal import from
`backend` exposes the unguarded writer, so a caller can persist an invalid graph
without bypassing Python visibility conventions.
