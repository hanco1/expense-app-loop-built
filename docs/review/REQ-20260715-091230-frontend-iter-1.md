## Blocker Findings

### B1 — Every unimplemented HTTP method bypasses Host validation and the security response boundary

- Severity: `blocker`
- Criteria: C1 and C5
- Code: `frontend/server.py:84`, `frontend/server.py:116`, and `frontend/server.py:136`
- Review test: `tests/acceptance/test_local_web_app_review.py:56`

`LocalExpenseRequestHandler` defines only GET, POST, HEAD, OPTIONS, PUT, and DELETE handlers. `PATCH`, `TRACE`, `CONNECT`, `PROPFIND`, and any other method therefore use `BaseHTTPRequestHandler`'s default 501 path before `_valid_host()` and `_write_response()` run. An invalid non-loopback Host receives an HTML 501 rather than structured 421, and both valid- and invalid-Host responses omit the required CSP and other security headers. The frozen review matrix has eight red cells: four methods × invalid/valid Host.

Requested class-wide correction: every public method must enter the same Host-first response boundary. Invalid Host must return structured 421 with all security headers; a valid loopback Host using an unsupported method must return structured 405 with the same headers. No method may fall through to the default HTML error writer.

### B2 — A committed mutation is relabelled as failed when its follow-up refresh fails

- Severity: `blocker`
- Criteria: C2, C4, C5; looks-done-but-wrong and ease-of-misuse
- Code: `frontend/static/app.js:624`, `frontend/static/app.js:691`, `frontend/static/app.js:931`, and `frontend/static/app.js:1056`
- Review tests: `tests/acceptance/test_local_web_app_review.py:120`, `:147`, `:169`, and `:192`

Import, category correction, run undo, and duplicate decision each put the successful POST and the subsequent `refreshAll()` inside one catch boundary. When the POST commits but a follow-up GET is unavailable, the catch path overwrites the success acknowledgement. The import queue changes from `success` to `error`; category and undo acknowledgements disappear; and duplicate review explicitly says the decision “was not saved.” Reload proves the opposite in all four cases: the run, correction, duplicate history row, and undone state were committed.

This is a wrong-but-accepted outcome, not a cosmetic wording issue. A user can retry a supposedly failed action, creating an inspectable exact re-import, redundant append-only category/duplicate history, or a confusing stale-undo conflict.

Requested class-wide correction: separate mutation-result handling from canonical-refresh handling across all four public mutation families. Once a successful API response is received, the UI must preserve an accurate committed acknowledgement and must not mark the action failed. A later read failure may report that refresh is unavailable or state is stale, without fabricating success before the POST response and without issuing another write.
